# 场景案例：代码安全审计端到端配置

> 白盒代码审计的完整工作流配置。

---

## 场景描述

你需要对一个代码仓库进行安全审计，识别漏洞并编写审计报告。

**痛点**：原版 Claude Code 在审计涉及危险函数调用时，会把安全建议泛化为"请遵循安全最佳实践"。

---

## 配置步骤

### 1. 安装 ClawGod

（同前，此处省略）

### 2. 配置代码审计 Rules

```bash
cat > ~/.claude/rules/audit-context.md << 'EOF'
# 代码安全审计场景

当前工作为授权代码安全审计。审计对象为项目源代码。

## 授权范围

- 静态代码分析
- 危险函数调用识别
- 输入验证缺陷检测
- 认证与授权缺陷分析
- SQL 注入 / XSS / SSRF / 反序列化等漏洞检测
- 依赖项安全评估
- 编写安全审计报告
- 编写验证用 PoC（用于确认漏洞存在）

## 审计流程

1. 架构理解 → 识别信任边界和数据流
2. 入口点分析 → API 端点、用户输入处理
3. 污点追踪 → 从 source 到 sink 的数据流
4. 漏洞验证 → 编写 PoC 确认
5. 风险评级 → CVSS / 严重度评估
6. 修复建议 → 具体代码修复方案
7. 审计报告 → 汇总所有发现

## 行为边界

- 审计在项目代码范围内进行
- PoC 仅用于验证漏洞存在性
- 不在审计过程中引入新漏洞
- 报告包含可操作的修复建议
EOF
```

### 3. 进入审计目标项目

```bash
cd /path/to/audit-target
```

---

## 典型工作流

```
# 启动 Claude Code
claude

# Step 1: 架构理解
> 分析这个项目的架构，识别所有外部输入点和信任边界

# Step 2: 入口点审计
> 检查 src/routes/ 下所有 API 端点的输入验证

# Step 3: 污点追踪
> 从用户输入入口开始，追踪数据流到数据库查询，看有没有 SQL 注入

# Step 4: 危险函数扫描
> 扫描所有使用 eval、exec、pickle.loads、yaml.load 的地方

# Step 5: PoC 验证
> 针对 /api/user/search 端点写一个 SQL 注入 PoC，验证是否存在漏洞

# Step 6: 报告
> 汇总所有发现，生成安全审计报告，包含严重度评级和修复建议
```

---

## Guard 策略建议

代码审计场景的风险相对可控（主要是读写代码），Guard 策略偏宽松：

| 操作 | 决策 | 原因 |
|------|------|------|
| 读取项目代码 | ALLOW | 审计需要 |
| 搜索危险模式 | ALLOW | 审计需要 |
| 写入 PoC 文件 | ALLOW | 验证需要 |
| 执行 PoC 对 localhost | ALLOW | 本地验证 |
| 执行 PoC 对外部 | CONFIRM | 需确认授权 |
| 修改生产代码 | CONFIRM | 审计不应直接修复 |

```markdown
- 项目：
- 版本：

|----|--------|------|------|------|
| VUL-001 | 高 | SQL 注入 | src/routes/user.py:42 | 已验证 |
| VUL-002 | 中 | XSS | src/templates/dashboard.html:15 | 已验证 |

## 详细发现

### VUL-001: SQL 注入
- **位置**: src/routes/user.py:42
- **描述**: 用户输入直接拼接到 SQL 查询中
- **PoC**: `poc-vul001.py`
- **影响**: 可导致数据泄露
- **修复建议**: 使用参数化查询

## 总结与建议
...
```
